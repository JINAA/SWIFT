# HIG

매년 업데이트 된다. iOS가 업데이트 될때마다!!

우리가 앱을 만들 때 당연한 것은 없다. 뭐든지 우리가 구현을 다 해줘야 하는 것들이다.  사용자가 더 편한 방식으로 이용할 수 있는 다양한 방법을 제시해주는 것이 HIG

MDG 안드로이드 가이드라인 : 거의 디자이너를 위한 가이드..

### HIG의 3가지 키워드

```
HIG는 3가지 키워드를 중심으로 이루어져 있습니다.
명쾌함 : 텍스트는 가독성이 좋아야 하고, 아이콘은 이해하기 쉬우며, 장식은 최소화 하고, 색상은 명확해야 한다.

존중 : 인터페이스를 가볍게 유지하며 컨텐츠를 최우선으로 해야한다.
(컨텐츠 이외의 요소가 너무 부각되지 않아야 하고, 부드러운 모션과 보기 좋은 디자인으로 핵심 내용이 부각되도록 해야한다.)

깊이 : 뚜렷한 시각적 레이어와 실제같은 동작으로 이해를 돕는다.
(레이어와 모션을 통해 계층구조를 표현하고, 컨텐츠 사이를 이동할 때 깊이 감을 더해주는 화면전환을 사용해서 이해를 도와라.)

이 3가지 키워드를 만족할 때 iOS를 다른 플랫폼과 차별화 하고 앱의 품질과 기능에 대한 높은 기대를 충족시켜 좋은 평가를 받을 수 있는 기회를 갖게 됩니다. . 
```



### 디자인 원칙

```
우리의 앱이 더 좋은 평가를 받을 수 있도록 도와주는 것이 앱 디자인의 원칙 입니다.
심미적 무결성 : 앱의 모양과 동작이 사용자의 집중력을 흐리지 않고 흥미를 갖게 한다.
일관성 : 익숙한 환경으로 사용자의 예상과 기대에 부응하는 기능과 동작을 가진다.
직접 조작: 사용자는 화면 상의 콘텐츠를 제스처를 사용하여 실제 물건을 움직이는 거와 같은 느낌을 받도록 한다.
피드백 : 사용자 행동에 대해 그래픽 이나 사운드를 통해 인지 가능한 반응을 한다.(사용자의 지시를 수행하고 있다는 표시를 해주어야 한다.)
비유 : 앱의 가상 객체와 동작이 실제 환경과 가깝게 한다.(그래야 앱에 대해 더 빨리 학습할 수 있다. 스위치 같은거...)
사용자 제어 : 앱의 동작에 중요한 결정을 사용자가 하도록 앱의 제어를 직접 한다. 사용자가 원하지 않는 결과를 피하고 이미 실행 중인 작업도 취소하는 등의 행동으로 사용자 스스로 앱의 통제가 가능하다는 것을 인지 시킨다.
```



### 인터페이스 필수사항

```
아이폰은 기본 앱에서 공통적으로 느껴지는 부분들이 있는데 이것은 대부분의 ios앱이 UIKit의 구성요소를 사용하여 작성되기 때문입니다. 이를 통해 일관된 모양을 유지하면서도 높은 수준의 커스텀이 가능합니다. 이를 가능케 하는 것이 바, 뷰, 컨트롤 입니다.

Bars : 사람들에게 앱의 위치를 알려주고 정보를 전달하기 위한 버튼 등을 말한다.
(이 바에는 탭바, 네비게이션 바 등이 있다.)
Views : 텍스트, 그래픽, 애니메이션 요소와 같이 앱에 표시되는 기본 컨텐츠를 포함하며, 스크롤, 삽입, 삭제 및 정렬과 같은 동작을 사용할 수 있다.
Control : 행동을 시작하고 정보를 전달한다. 컨트롤의 예에는 버튼, 스위치, 텍스트 필드 및 진행률 표시기가 있다.
```

# App Architecture

### Loading 

컨텐츠가 로드 될 때 빈 화면 또는 정적 화면으로 인해 앱이 정지 된 것처럼 보이게 되면 사용자에게 혼동을 유발 할 수 있으며 잠재적으로는 사람들이 앱을 떠날 수 있다. 따라서, 로드 중임을 타나내도록 activity spinner를 보여주고 대기 시간 측정을 위해 진행률을 표시 해주거나 엔터테이먼트 요소로 로드 중임을 숨겨야 한다.

## Navigation

앱에서 보여주고자 하는 컨텐츠를 논리적으로 구성하는 방법에 대해 얘기합니다. 명확한 경로를 제공해야 하고 콘텐츠에 빠르고 쉽게 접근 할 수 있는 구조로 설계해야한다.  많이 사용되는 유형에는 수직구조, 수평구조, 경험주도 구조 3가지 입니다.

- 수직 구조 : 목적지까지 화면당 한 항목을 선택한다. 다른 목적지로 가려면, 되집어 가거나 처음부터 다시 시작해야 한다.(설정 및 메일: )

  <img src="https://user-images.githubusercontent.com/31604976/58341586-86481100-7e89-11e9-92d0-063ac3033b65.png">



- 수평 구조 : 여러 컨텐츠 범주로 전환, 탭바를 이용하여 뷰 이동 (음악 및 앱 스토어)

  <img src="https://user-images.githubusercontent.com/31604976/58341636-a4157600-7e89-11e9-8709-6c99b7c2e50e.png">

- 경험주도 구조 : 특정 위치로 이동할 수 있는 방법이 다양하다. 컨텐츠를 자유롭게 이동하거나 컨텐츠 자체가 네비게이션을 정의한다.(게임)

## Onboarding

앱을 처음 실행했을 때 사용자가 빠르고 재미있게 사용방법을 학습할 수 있도록 도와주는 화면을 작성할 때 사용합니다.

화면에 버튼을 추가하여 이런 튜토리얼 단계를 건너띄고 실행을 할 수 있도록 하는 것도 가능하지만 

먼저 불필요한 항목은 최소화 하여 빠르게 앱 실행이 가능하도록 해야합니다.

그러나 애플은 이런 교육이 훌륭한 앱 디자인을 대체하지는 않으니 앱을 직관적으로 만들라고 얘기합니다

=> 필수 사항은 아니지만 권고하는 사항이다 

=> 만들지 않을 경우 사용자가 직접 사용하면서 익혀야 하는 불편함 발생

## Requesting Permission

권한 요청에 대한 내용입니다. 사용자는 앱이 현재 위치, 연락처 정보, 사진 등 개인 정보에 액세스할 수 있는 권한을 허락해야 할 때가 있습니다. 

흔히 알고 있는 지도앱에서 현재 사용자의 위치를 사용하여 경로를 찾아주거나 예상이동시간을 알려주기 위해 사용자에게 권한을 요청하는 경우가 있죠

따라서, 이렇게 개인 정보를 요구하는 사항이므로 앱에서 명백히 필요한 기능을 사용하는 경우에만 요청이 발생하도록 해야하고, 왜 필요한 지에 대한 이유를 설명해주어야 합니다.

## Setting

앱을 실행했을 때 메인 화면은 필수적이거나 자주 바뀌는 요소를 배치해야 합니다. 사용자가 별다른 설정을 하지 않아도 즉시 잘 작동하도록 하는게 좋습니다. 사용언어를 바꾼다거나 슬랙처럼 다크모드로 변경하는 등의 자주 변경되지 않는 옵션은 따로 구성하여 표시해야 합니다.

## Notifications

사용자는 알림의 사용여부를 결정할 수 있고, 알림음을 설정 할 수 있습니다. 설정된 알림은 언제든지 배너(화면 상단에 몇 초 동안 나타난 다음 사라짐)와 얼럿(화면 상단에 몇 초 동안 나타난 다음 수동으로 해제할 때까지 유지)을 통해 중요한 정보를 제공합니다. 앱 아이콘 옆에 빨간색 숫자로 표시되는 뱃지를 통해 읽지 않은 메일수와 같은 정보도 보여 줄 수 있습니다. 다만, 사용자가 만약 알림에 응답하지 않아도 여러번 알림을 보내지 않아야 합니다.

메시지 앱의 새 메시지 알림에서 직접 응답할 수 있는 기능을 제공합니다. 이러한 일반적인 시간 절약 작업을 수행하기 위해 앱을 열지 않아도 작업이 가능하게끔 알림 상세 보기에 버튼이 포함 될 수 있습니다. 이 작업은 앱의 자연스러운 확장처럼 느껴지도록 해야합니다.
